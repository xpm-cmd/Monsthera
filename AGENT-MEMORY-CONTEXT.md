## Agora Consensus System Architecture

### Verdict Storage & Retrieval

**Database Schema (review_verdicts table):**
- id: primary key
- ticketId: foreign key to tickets
- agentId: string (agent submitting verdict)
- sessionId: string (agent's session)
- specialization: string (council role: architect, security, simplifier, etc.)
- verdict: string (pass/fail/abstain)
- reasoning: text (optional)
- createdAt: timestamp
- supersededBy: integer (points to new verdict ID if replaced)

**Verdict Chain Logic:**
When a NEW verdict is submitted for the same (ticketId, specialization) pair:
1. Query finds existing row where (ticketId=X AND specialization=Y AND supersededBy IS NULL)
2. Sets supersededBy = -1 (placeholder)
3. Inserts new verdict with supersededBy=NULL
4. Updates old verdict with supersededBy = new_row.id
This creates an immutable audit chain.

**Active Verdict Query (getActiveReviewVerdicts):**
- Fetches all rows WHERE (ticketId=X AND supersededBy IS NULL)
- Orders by createdAt then id
- Returns entire set of "latest" verdicts per specialization

### Consensus Verdict Selection

**Key Finding: Per-Specialization Latest Selection**
In buildConsensusPayload():
1. Input: verdictRows = array of active verdicts (potentially from multiple agents per specialization)
2. For EACH councilSpecialization in the council:
   - Use `.find()` to get THE FIRST matching verdict from verdictRows
   - This returns ONLY ONE verdict per specialization
3. If multiple agents have submitted verdicts for same specialization:
   - Only the FIRST one in the verdictRows array is used
   - verdictRows is ordered by createdAt, id (oldest first)
   - Therefore: EARLIEST submitted verdict per specialization wins

**This is the critical behavior:**
```typescript
const verdicts = councilSpecializations.flatMap((specialization) => {
  const verdict = normalizedVerdicts.find((entry) => entry.specialization === specialization);
  return verdict ? [{ ...verdict }] : [];
});
```

The `.find()` returns first match only. With multiple agents submitting for same specialization, oldest by createdAt wins.

### Consensus Evaluation Process

**buildConsensusPayload() outputs:**
1. verdicts: array (one per specialization, oldest per spec if multiple agents voted)
2. counts.pass: number of "pass" verdicts
3. counts.fail: number of "fail" verdicts
4. counts.abstain: number of "abstain" verdicts
5. counts.responded: total verdicts included
6. counts.missing: specializations still without verdicts
7. quorumMet: (counts.pass >= requiredPasses)
8. blockedByVeto: (any veto from architect or security)
9. advisoryReady: (quorumMet AND !blockedByVeto AND !diversityBlocked)

**Veto Mechanism:**
- Architect and Security are default veto specializations
- If ANY veto specialization returns "fail", blockedByVeto=true
- advisoryReady becomes false even if quorum is met

### Auto-Advance Trigger

**Location: submit_verdict tool (in src/tools/ticket-tools.ts)**

When a verdict is submitted:
1. Call insertReviewVerdict() to store it
2. Call getActiveReviewVerdicts() to get current state

4. IF consensus.advisoryReady AND governance.autoAdvance !== false:
   - Look up GATED_ADVANCE_TARGET for current ticket status

**Also in check_consensus tool:**

### Governance Options (Non-voting Roles)

**buildGovernanceOptions() filters verdicts:**
- Takes config.governance.nonVotingRoles
- Maps to agent roleIds (facilitator, etc.)
- Builds nonVotingAgentIds set
- In buildConsensusPayload(), verdicts from non-voting agents are excluded from quorum count
- They appear in governance.nonVotingExcluded but don't contribute to pass/fail/abstain counts
- Council specializations are narrowed to GOVERNANCE_ANALYTICAL_SPECIALIZATIONS (5 roles) instead of all 6

### Model Diversity (strictDiversity option)

When governance.modelDiversity.strict=true:
1. evaluateModelDiversity() analyzes pass verdicts by agent identity
2. Groups by (provider, model) pair
3. If any model appears in multiple specializations → duplicateGroups
4. If duplicateGroups.length > 0 OR missing identities → diversityMet=false
5. countDiversityAdjustedPasses() counts unique models only (not total pass verdicts)
6. diversityBlocked blocks advisoryReady even if quorum met

### Gated Transitions

Only two transitions are gated with quorum enforcement:
1. technical_analysis→approved
2. in_review→ready_for_commit

All other transitions can be done without council consensus (though advisors are still polled)

### suggest_next_work_logic

## suggest_next_work Tool Implementation (src/tools/read-tools.ts)

### Purpose
Suggest approved tickets that match the developer's claimed files. Returns ranked list by path overlap + priority.

### Execution Flow
1. **Load claimed files**: Parse session.claimedFilesJson (max 50 items)
2. **Load approved unassigned tickets**: queries.getTicketsByRepo(db, repoId, {status: "approved"})
3. **Score tickets** by:
   - Count overlapping paths: claimed.filter(cp => affected.some(ap => pathsOverlap(cp, ap)))
   - rankingScore = overlapPaths.length * 100 + ticket.priority
   - Higher overlap weighted 100x more than priority difference
4. **Sort by rankingScore DESC, then ticketId ASC**
5. **Determine match kind**:
   - "clear_match": single ticket with highest score
   - "ambiguous_match": multiple tickets tied for top score
   - "no_match": no overlap, sorted by priority only
6. **Return routing recommendation**:
   - high confidence: clear_match with path overlap
   - medium confidence: priority_only match (no claimed files yet)
   - low confidence: ambiguous/no match

### Path Overlap Detection
```typescript
pathsOverlap(left, right) {
  const a = normalizeClaimPath(left)  // trim, strip ./, trailing /
  const b = normalizeClaimPath(right)
  return a === b || a.startsWith(`${b}/`) || b.startsWith(`${a}/`)
}
```
Matches exact paths or parent/child relationships.

### Output
JSON object with:
- match: {kind, reason}
- routingRecommendation: {action, ticketId?, ticketIds?, confidence, basis?}
- suggestions: ranked list of tickets with matchKind + reason for each
- totalApprovedUnassigned: count

### Database Queries
- queries.getSession(db, sessionId) → {claimedFilesJson, ...}
- queries.getTicketsByRepo(db, repoId, {status: "approved"}) → Ticket[]
- Ticket fields used: ticketId, title, status, severity, priority, affectedPathsJson

### 3. QUORUM CHECKPOINT (lines 275-513)

This is a specialized step for consensus-based approval gates:
- **executeQuorumCheckpoint()** executes:
  1. Initial consensus check on loaded verdicts
  2. If not ready: dispatch reviewer resolution via runtime.resolveReviewers hook
  3. Send coordination messages to resolved agents
  4. Poll loop: repeatedly checks verdicts every pollIntervalMs until:
     - Consensus ready (all roles passed/abstained)
     - Veto encountered (architect or security vetoes)
     - Quorum blocked (not enough passes)
     - Timeout reached
  5. Returns output with consensus state, roles, dispatch report, checkpoint status

Key constants:
- DEFAULT_QUORUM_ROLES = analytical specializations [architect, simplifier, security, performance, patterns, design]
- DEFAULT_QUORUM_POLL_INTERVAL_MS = 2000

  - Context keys: params, steps (step output map), last, item, itemIndex

- **resolvePath()** (lines 573-578): Dot notation path navigation
  - Example: "steps.ticket.affectedPaths.0" → context.steps.ticket.affectedPaths[0]

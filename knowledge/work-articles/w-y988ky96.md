## Objective

Add a template-opt-in guard that blocks the `enrichment -> implementation` transition unless a fresh environment snapshot exists for the work article and the sandbox lockfiles match HEAD. This turns the snapshot MVP into something that actually gates bad handoffs instead of just decorating context packs.

## Context

Research note `k-to46fuoi` argued for smarter guards as one of the three main pay-offs of the snapshot work. Parent work article `w-0ieze72s` (PR #59, commit `aafe134`) shipped record / get_latest / compare plus the context-pack integration, but guards still only look at `WorkArticle` shape — they never consult the physical sandbox. A reviewer resuming a work article can advance it to `implementation` on a stale sandbox with drifted lockfiles today.

The guard design is opt-in: we do not want to break the spike / bugfix / refactor flows that never needed a snapshot. Feature templates that call out snapshot-awareness are the target; all other templates keep the current guard set.

## Scope

In scope:

- New guard `snapshot_ready` in `src/work/guards.ts`, or a separate file if it keeps the async/service dependency out of the pure-function file. Rules:
  - a snapshot exists for `workId`,
  - `ageSeconds < maxAgeMinutes * 60` (use the existing `SnapshotService.maxAgeMinutes`, i.e. `MONSTHERA_SNAPSHOT_MAX_AGE_MINUTES`, default 30),
  - every `lockfile.sha256` matches the current HEAD lockfile hash. We compute HEAD hashes at call time (not inside the guard set, to keep the fn dependency-free).
- Opt-in plumbing: `WorkTemplateConfig` gains `requiresSnapshotForImplementation?: boolean`. Default `false`. Documented on the type.
- Template update: `FEATURE` template sets `requiresSnapshotForImplementation: true`. `BUGFIX` / `REFACTOR` / `SPIKE` stay `false`.
- Guard injection: extend `getGuardSet` to accept an optional deps bag `{ snapshotService, headLockfileHashes }` so the async snapshot check is performed at the service boundary (inside `checkTransition` / `advancePhase`) rather than in the pure guard functions. Keep the synchronous `check(article) => boolean` shape for existing guards.
- Async guard variant: introduce a `AsyncGuardEntry` that coexists with the existing sync ones. `checkTransition` evaluates sync guards first, then awaits the async ones. Service wiring passes `snapshotService` only when it is available.
- `WorkService.advancePhase` reads the lockfile hashes of HEAD (read from disk once per call, not per guard) and passes them into `checkTransition`.
- Tests:
  - New guard unit test for the three acceptance rules.
  - Lifecycle test: feature article with fresh snapshot + matching lockfiles advances; stale snapshot, missing snapshot, or mismatched lockfile hash fails with `GuardFailedError`.
  - Backwards compat test: bugfix / refactor / spike transitions keep passing without a snapshot.
  - Existing lifecycle tests keep passing unchanged (no regression on template-default behavior).
- ADR documenting the opt-in design. File: `docs/adrs/006-snapshot-ready-guard.md`. Why it is opt-in, how to turn it on for a template, and how `skipGuard` interacts with it (same as any other guard — auditable bypass).

Out of scope:

- Changing the `InMemorySnapshotRepository`.
- Moving `src/context/`.
- Dashboard surfacing of snapshot-ready status. (Covered by follow-up 3.)
- Benchmarking. (Follow-up 4.)
- Exposing a new MCP tool. The guard is a server-side predicate, not an agent-callable tool.

## Constraints

- The MCP server still runs no shell. Lockfile hashes are read off the working tree with `node:crypto` / `node:fs`, not via a subprocess.
- Do NOT break the sync `GuardEntry` contract. Async guards are additive.
- Do NOT hardcode lockfile paths. Read from `WorkArticle.codeRefs` if applicable, otherwise default to a small allowlist of well-known names (`pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `uv.lock`, `poetry.lock`, `Cargo.lock`, `go.sum`) — only those that exist on disk are hashed.
- Do NOT modify `InMemorySnapshotRepository` or move `src/context/`.
- `skipGuard` stays the only bypass. No template-level disable flag.

Created:

- `src/work/lockfile-hashes.ts` — `readHeadLockfileHashes` walks a conservative allowlist (`pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `uv.lock`, `poetry.lock`, `Cargo.lock`, `go.sum`) on disk using `node:fs` + `node:crypto`. Missing / unreadable files skip silently — the sandbox is allowed a subset.
- `tests/unit/work/snapshot-ready-guard.test.ts` — 9 unit tests over the async guard: opt-in scoping, fail-closed paths (no snapshot, stale, mismatched hash, missing lockfile entry), pass paths, `maxAgeMinutes=0` disabling freshness but still enforcing lockfiles, `skipGuard` bypass recording the guard name.
- `tests/unit/work/snapshot-ready-service.test.ts` — 6 end-to-end tests exercising the guard through `WorkService.advancePhase` with real lockfiles on disk and a wired `SnapshotService`.
- `docs/adrs/006-snapshot-ready-guard.md` — ADR documenting the opt-in design, the two-pass guard model (sync then async), the fail-closed-on-missing-deps rationale, and `skipGuard` interaction.

Modified:

- `src/work/guards.ts` — added `snapshot_ready(article, deps)` alongside the existing pure guards, plus a local `SnapshotGuardDeps` type so guards.ts stays a leaf file (no import from lifecycle.ts).
- `src/work/lifecycle.ts` — added `AsyncGuardEntry`, `GuardDeps`, `getAsyncGuardSet`, and `evaluateAsyncGuards`. `checkTransition` and the existing sync path are untouched. Async evaluator mirrors sync behavior: `skipGuard` bypasses failures, the guard name lands in `skippedGuards`.
- `src/work/templates.ts` — `WorkTemplateConfig.requiresSnapshotForImplementation?: boolean`. Only `FEATURE` opts in; bugfix / refactor / spike stay false.
- `src/work/repository.ts` — `AdvancePhaseOptions.guardDeps?: GuardDeps` (per-call, not per-repo). `advancePhase` return type widened to include `GuardFailedError` to match reality.
- `src/work/file-repository.ts` + `src/work/in-memory-repository.ts` — `advancePhase` runs the sync `checkTransition` first, then awaits `evaluateAsyncGuards`, concatenating `skippedGuards` from both passes before writing phase history.
- `src/work/service.ts` — `WorkServiceDeps` gains optional `snapshotService` + `repoPath`. New private `enrichOptionsWithGuardDeps` computes the guard deps (snapshot service + pre-hashed HEAD lockfiles) only when the template opts in, attaches them to `AdvancePhaseOptions.guardDeps`, and delegates.
- `src/core/container.ts` — reorders `snapshotService` creation ahead of `WorkService` and passes it + `config.repoPath` into the service constructor. No behavior change to the snapshot service itself or to other services.
- `CHANGELOG.md` — new `Changed` bullet under `[Unreleased]` describing the opt-in guard and referencing the ADR.

Key design choices (captured in ADR-006):

- Fail-closed on missing deps. `getAsyncGuardSet` returns an empty array when `deps?.snapshotService` is not supplied. Rationale: unit tests and repo-level callers that predate this feature must not regress; production wires the service, so the guard always fires there.
- No mutation to the sync guard contract. Async guards are additive, evaluated as a second pass. Engineers adding a new guard have a clear place for service-backed ones without risk of async sneaking into the sync path.
- Per-call deps, not per-repo. Repos stay stateless w.r.t. the snapshot service — the `WorkService` decides per call whether the template requires deps and attaches them.
- Lockfile hashing uses `node:fs` + `node:crypto`. No subprocess. Consistent with the MCP-server-never-shells-out rule that PR #59 preserved.

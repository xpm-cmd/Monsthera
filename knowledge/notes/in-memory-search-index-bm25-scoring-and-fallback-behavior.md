## Overview

`InMemorySearchIndexRepository` is the fallback search implementation used when Dolt is unavailable. It provides BM25-based text search and cosine-similarity semantic search entirely in-process, with no external dependencies. It lives in `src/search/in-memory-repository.ts`.

## When It Activates

The container (`src/core/container.ts`) uses this class in two scenarios:
1. **Dolt is not enabled** (`config.storage.doltEnabled` is false) — the default for single-user local setups.
2. **Dolt failed to initialize** — connection or schema errors cause a graceful fallback. The status check reports `healthy: false` with `"degraded — Dolt unavailable"`.

In both cases, `InMemoryOrchestrationEventRepository` is also used for the event store.

## Architecture

The repository maintains four in-memory data structures:
- `documents: Map<string, IndexedDocument>` — full document store (id, title, content, type).
- `invertedIndex: Map<string, Set<string>>` — term-to-document-IDs mapping for BM25.
- `titleTerms: Map<string, Set<string>>` — per-document title term sets for boosting.
- `embeddings: Map<string, number[]>` — optional embedding vectors for semantic search.

A `dirtyIds: Set<string>` tracks documents modified since last reindex to enable incremental rebuilds.

## BM25-lite Scoring

The `bm25Score()` method implements a simplified BM25 without document-length normalization:
- **Term frequency saturation**: `tf / (tf + k1)` where `k1 = 1.2`.
- **IDF**: `log((N - df + 0.5) / (df + 0.5) + 1)` — standard BM25 IDF formula.
- **Title boost**: terms appearing in the document title get a `3x` multiplier, making title matches dominate.
- No average document length (`avgdl`) or `b` parameter — this simplification works because Monsthera articles are relatively uniform in length.

## Incremental vs Full Reindex

The `reindex()` method is smart about rebuilding:
- **No-op**: if no documents or no dirty entries with a populated index.
- **Incremental** (< 50% dirty): only removes and re-tokenizes the dirty documents.
- **Full rebuild** (>= 50% dirty or empty index): clears everything and re-tokenizes all documents.

This means adding a few articles after startup triggers fast incremental updates, while bulk imports trigger a single full rebuild.

## Semantic Search

When embeddings are stored (via `storeEmbedding()`), the `searchSemantic()` method performs brute-force cosine similarity across all stored vectors. This is a linear scan — fine for the expected knowledge base size (hundreds to low thousands of articles) but would need an approximate nearest-neighbor index for larger scales.

## Snippet Generation

Search results include context snippets generated by `generateSnippet()`:
- Finds the first occurrence of any query term in the content.
- Extracts 80 characters of context on each side of the match.
- Falls back to the first 160 characters if no term matches in the body (match was in the title only).

## Canary Check

The `canary()` method verifies index health by picking the first term from the inverted index and confirming that searching for it returns at least one result. This is used by the status/health system to detect index corruption.

## Trade-offs vs Dolt

| Aspect | In-Memory | Dolt |
|--------|-----------|------|
| Persistence | Lost on restart (rebuilt from Markdown) | Survives restarts |
| Startup cost | Must reindex all articles | Reads from existing tables |
| Dependencies | None | MySQL-compatible server |
| Concurrency | Single-process only | Multi-process safe |
| Semantic search | Brute-force cosine | Same (no ANN in either) |
